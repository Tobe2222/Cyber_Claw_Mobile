## 2026-07-22 14:58 (channel: #cyber-dev)
- **User:** Tobe reported "i also tried to update/edit a quest but nothing happened and this error appeared" — screenshot showed the Edit quest modal with "CYBERHIVE_WEBSITE V3" filled in and after tapping Save, a bottom-of-screen error: "Couldn't update quest: quest not found". Also: "i could not add or edit the steps and goal" — that's a pre-existing v3.10.x design limitation (the editor doesn't yet support goal text editing; the screenshot shows the "Goal text editing lands in a future release" hint), out of scope for this fix.
- **You:** v3.10.73 fixes the "quest not found" error. Root cause: a cache-vs-broadcast race in QuestsScreen. The screen hydrates from AsyncStorage cache for instant paint, then fires `requestQuestsList()` to get a fresh broadcast ~100ms later. If the user taps ✏️ within that window, the editor opens with a copy of the cached quest — including its id. When the desktop has a different id for the same quest (after desktop reinstalls, manual file edits, or older cache data), the editor's id doesn't match what the desktop has, and Save → desktop's `onUpdateQuest` → `findIndex < 0` → returns null → ack "quest not found". Tobe hit this in a session where the cached ids and desktop ids diverged. Fix: new `firstBroadcastReceived` state, reset to false on every screen mount and set to true in the broadcast handler. ✏️, ☆, and ✕ buttons are gated + dimmed (opacity 0.35) until first broadcast arrives. A "⏳ Syncing with desktop..." hint shows above the card list. Once the broadcast lands (typically <100ms), buttons enable. +New button unaffected (creates a new quest, desktop assigns fresh id). Bumped to v3.10.73 (versionCode 299). Pushed to origin. See CHANGES_3.10.73.md for the breakdown and the lesson "cache + live broadcast is a stampede zone — any state populated from two sources has a window where the values can disagree; the fix is to gate destructive actions on the live broadcast arriving".
