## 2026-07-22 14:42 (channel: #cyber-dev)
- **User:** Tobe reported "that bottom gap appears after i have opened keyboard and then removed it again" — key detail: the gap only appears AFTER keyboard interaction, not always (so the v3.10.70 nav-bar inset fix was correct but didn't cover this case).
- **You:** v3.10.71 fixes the keyboard-dismiss gap by changing `<KeyboardAvoidingView behavior='padding'>` to `<KeyboardAvoidingView behavior='padding' enabled={Platform.OS === 'ios'}>`. When `enabled={false}`, the component renders as a plain `<View>` and does nothing — no event listeners, no padding adjustments. On Android, `adjustResize` in AndroidManifest.xml already handles keyboard avoidance natively (window shrinks → flex-end inputContainer pushes up; window grows back → layout returns). On iOS we keep KeyboardAvoidingView because there's no native adjustResize equivalent. The v3.10.70 nav-bar inset fix (paddingBottom: 8 + insets.bottom) is still correct — it handles the always-present inset, this handles the post-keyboard-dismiss gap. They're orthogonal. Bumped to v3.10.71 (versionCode 297). Pushed branch + tag to origin. See CHANGES_3.10.71.md for the breakdown and the lessons "when the user describes WHEN a bug appears, that's the debugging fingerprint" + "KeyboardAvoidingView behavior='padding' on Android is a known footgun" + "don't stack two keyboard-avoidance systems".
